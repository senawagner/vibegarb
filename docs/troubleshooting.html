🛠️ Troubleshooting - Vibe Garb E-commerce

Guia completo de resolução de problemas

📋 Visão Geral

Este documento registra todos os problemas encontrados durante o desenvolvimento do Vibe Garb e suas respectivas soluções. Serve como referência para futuras implementações e manutenção.

🚨 Problema 1: Conflitos de Porta Docker

❌ Sintomas

  • Erro: port is already allocated
  • Containers não iniciam
  • Porta 8500 já em uso

🔍 Diagnóstico

# Verificar portas em uso
netstat -ano | findstr :8500
netstat -ano | findstr :3350
netstat -ano | findstr :6350

✅ Solução

1. Parar containers conflitantes:

# Parar todos os containers
docker stop $(docker ps -a -q)

# Ou parar containers específicos
docker stop container_name

2. Configurar portas únicas no docker-compose.yml:

services:
  webserver:
    ports:
      - "8500:80"  # Porta única para Vibe Garb
  db:
    ports:
      - "3350:3306"  # MySQL único
  redis:
    ports:
      - "6350:6379"  # Redis único

3. Reiniciar com novas configurações:

docker-compose down
docker-compose up -d

📝 Prevenção

  • Usar portas únicas para cada projeto
  • Documentar portas utilizadas
  • Verificar disponibilidade antes de subir containers

🚨 Problema 2: Timeout de Assets (504 Gateway Timeout)

❌ Sintomas

  • Erro 504 ao carregar CSS/JS
  • Página carrega mas sem estilos
  • Console mostra timeout em assets

🔍 Diagnóstico

# Verificar se arquivo hot existe
ls -la public/hot

# Verificar logs do Nginx
docker logs vibegarb_webserver

✅ Solução

1. Remover arquivo hot (se existir):

rm -f public/hot

2. Rebuild dos assets:

docker exec vibegarb_node npm run build

3. Limpar cache do Laravel:

docker exec vibegarb_app php artisan optimize:clear

📝 Prevenção

  • Não usar modo hot em produção
  • Sempre fazer build antes de deploy
  • Verificar se assets foram compilados corretamente

🚨 Problema 3: Migração Vue→Blade (Arquitetura)

❌ Sintomas

  • Complexidade desnecessária com SPA
  • Performance lenta para SEO
  • Overhead de JavaScript desnecessário

🔍 Diagnóstico

Projeto inicialmente planejado com Vue.js SPA, mas para MVP com dropshipping, a complexidade era desnecessária.

✅ Solução

1. Migrar para Blade Templates:

# Estrutura final
resources/views/
├── layouts/
│   └── app.blade.php
├── products/
│   ├── index.blade.php
│   └── show.blade.php
└── cart/
    └── index.blade.php

2. Manter JavaScript mínimo:

// Apenas interações essenciais
- Adicionar ao carrinho
- Atualizar quantidade
- Busca em tempo real
- Filtros dinâmicos

3. Benefícios alcançados:

  • ✅ SEO otimizado (server-side rendering)
  • ✅ Performance melhorada
  • ✅ Manutenibilidade simplificada
  • ✅ Carregamento mais rápido

🚨 Problema 4: Performance Lenta no Windows

❌ Sintomas

  • Carregamento de páginas lento
  • Docker containers lentos
  • I/O de arquivos muito lento

🔍 Diagnóstico

# Verificar performance dos volumes
docker stats

# Verificar uso de CPU/memória
docker system df

✅ Solução

1. Otimizar volumes no docker-compose.yml:

services:
  app:
    volumes:
      - ./:/var/www:cached  # Usar :cached para performance
      - /var/www/vendor     # Excluir vendor do volume
      - /var/www/node_modules # Excluir node_modules

2. Configurar WSL2 (se aplicável):

# No .wslconfig
[wsl2]
memory=4GB
processors=4
swap=2GB

3. Otimizar Laravel:

# Aplicar otimizações
docker exec vibegarb_app php artisan optimize
docker exec vibegarb_app php artisan config:cache

📝 Resultado

  • ✅ Tempo de carregamento reduzido de 3s para < 1s
  • ✅ Containers mais responsivos
  • ✅ I/O de arquivos otimizado

🚨 Problema 5: Implementação Incorreta de Frontend (Vue.js vs. Blade)

❌ Sintomas

  • Erro ERR_EMPTY_RESPONSE ou 404 Not Found ao acessar a loja.
  • Erros no console do navegador relacionados ao manifesto do Vite (ex: Unable to locate file in Vite manifest: resources/js/Pages/Home.vue).
  • A página não carregava, indicando um erro fatal no backend durante a renderização.

🔍 Diagnóstico

Ao solicitar uma atualização de estilo na página inicial, a implementação foi feita de forma incorreta, criando um componente Vue.js (resources/js/Pages/Home.vue) em um projeto cuja arquitetura é estritamente baseada em Blade. O sistema tentou carregar e renderizar um asset Vue que não deveria existir, causando um conflito irrecuperável no processo de build e na renderização da view pelo Laravel.

✅ Solução

1. Remoção do Componente Vue Incorreto:

# Excluir o arquivo Vue que foi criado erroneamente
del resources\\js\\Pages\\Home.vue

2. Garantir que o Controller Retorne a View Blade:

Verificar se o HomeController e outras rotas estavam retornando a view Blade corretamente, sem referências a Inertia ou renderização de componentes Vue.

// Exemplo correto no Controller
return view('home', $data);

3. Rebuild dos Assets e Limpeza de Cache:

# Recompilar os assets (CSS/JS) apenas para o ambiente Blade
docker exec vibegarb_node npm run build

# Limpar todo o cache de configuração, rotas e views do Laravel
docker exec vibegarb_app php artisan optimize:clear

📝 Prevenção

  • Sempre validar a arquitetura do projeto (definida na documentação) antes de implementar novas features ou alterações.
  • Manter a separação clara: Blade para renderização de HTML no servidor e Vanilla JS para interatividade no cliente.
  • Evitar introduzir novas tecnologias (como Vue/React) sem uma discussão e planejamento prévio.

🚨 Problema 5: CSS Quebrado (Tailwind não carregando)

❌ Sintomas

  • Páginas sem estilos
  • Classes Tailwind não aplicadas
  • CSS não compilado

🔍 Diagnóstico

# Verificar se assets foram compilados
ls -la public/build/

# Verificar se Vite está funcionando
docker exec vibegarb_node npm run dev

✅ Solução

1. Reinstalar dependências:

# Limpar node_modules
docker exec vibegarb_node rm -rf node_modules
docker exec vibegarb_node npm install

2. Rebuild dos assets:

# Build de produção
docker exec vibegarb_node npm run build

# Verificar se arquivos foram gerados
ls -la public/build/assets/

3. Verificar configuração do Vite:

// vite.config.js
export default defineConfig({
    plugins: [
        laravel({
            input: ['resources/css/app.css', 'resources/js/app.js'],
            refresh: true,
        }),
    ],
});

🚨 Problema 6: Configuração .env (Credenciais MySQL)

❌ Sintomas

  • Erro de conexão com banco
  • Laravel não consegue conectar ao MySQL
  • Migrations falham

🔍 Diagnóstico

# Testar conexão com banco
docker exec vibegarb_app php artisan tinker
>>> DB::connection()->getPdo();

✅ Solução

1. Configurar .env corretamente:

# .env
DB_CONNECTION=mysql
DB_HOST=db
DB_PORT=3306
DB_DATABASE=vibegarb
DB_USERNAME=vibegarb
DB_PASSWORD=vibegarb123

# Configurações de cache
CACHE_DRIVER=redis
SESSION_DRIVER=file
QUEUE_CONNECTION=sync

2. Verificar se banco está rodando:

# Verificar containers
docker-compose ps

# Conectar ao MySQL
docker exec -it vibegarb_db mysql -u vibegarb -p

3. Executar migrations:

docker exec vibegarb_app php artisan migrate

🚨 Problema 7: Chave APP_KEY (Criptografia)

❌ Sintomas

  • Erro: No application encryption key has been specified
  • Sessões não funcionam
  • Autenticação falha

🔍 Diagnóstico

# Verificar se APP_KEY existe
echo $APP_KEY

# Verificar .env
cat .env | grep APP_KEY

✅ Solução

1. Gerar nova chave:

docker exec vibegarb_app php artisan key:generate

2. Verificar se foi gerada:

# Deve gerar algo como:
APP_KEY=base64:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

3. Limpar cache:

docker exec vibegarb_app php artisan config:clear

🚨 Problema 8: Extensão Redis PHP

❌ Sintomas

  • Erro: Class 'Redis' not found
  • Cache não funciona
  • Sessões não persistem

🔍 Diagnóstico

# Verificar extensões PHP
docker exec vibegarb_app php -m | grep redis

# Testar conexão Redis
docker exec vibegarb_app php -r "var_dump(extension_loaded('redis'));"

✅ Solução

1. Instalar extensão no Dockerfile:

# Dockerfile
RUN apt-get update && apt-get install -y \
    libredis-dev \
    && pecl install redis \
    && docker-php-ext-enable redis

2. Rebuild da imagem:

docker-compose down
docker-compose build --no-cache
docker-compose up -d

3. Verificar instalação:

docker exec vibegarb_app php -m | grep redis

🚨 Problema 9: Permissões Windows (Volumes Docker)

❌ Sintomas

  • Erro de permissão ao escrever logs
  • Laravel não consegue criar arquivos
  • Cache não funciona

🔍 Diagnóstico

# Verificar permissões
docker exec vibegarb_app ls -la storage/

# Tentar criar arquivo de teste
docker exec vibegarb_app touch storage/test.txt

✅ Solução

1. Ajustar permissões:

# Dentro do container
docker exec vibegarb_app chown -R www-data:www-data storage/
docker exec vibegarb_app chmod -R 775 storage/

2. Configurar volumes otimizados:

# docker-compose.yml
services:
  app:
    volumes:
      - ./:/var/www:cached
      - /var/www/storage  # Volume separado para storage

3. Verificar funcionamento:

# Testar escrita
docker exec vibegarb_app php artisan cache:clear

🚨 Problema 10: Rota Dashboard (Redirecionamento)

❌ Sintomas

  • Erro 404 ao acessar /dashboard
  • Usuário logado não é redirecionado
  • Rota não encontrada

🔍 Diagnóstico

# Verificar rotas
docker exec vibegarb_app php artisan route:list

# Verificar se middleware auth está funcionando
docker exec vibegarb_app php artisan route:list --name=dashboard

✅ Solução

1. Configurar rota no web.php:

// routes/web.php
Route::middleware(['auth'])->group(function () {
    Route::get('/dashboard', function () {
        return view('dashboard');
    })->name('dashboard');
});

2. Criar view dashboard:

// resources/views/dashboard.blade.php
@extends('layouts.app')

@section('content')


    
Dashboard

    
Bem-vindo, {{ Auth::user()->name }}!



@endsection

3. Configurar redirecionamento pós-login:

// app/Providers/RouteServiceProvider.php
public const HOME = '/dashboard';

🚨 Problema 11: Tabela Cache (Migrations)

❌ Sintomas

  • Erro ao executar migrations
  • Tabela cache não existe
  • Laravel não consegue criar tabelas

🔍 Diagnóstico

# Verificar status das migrations
docker exec vibegarb_app php artisan migrate:status

# Verificar se banco existe
docker exec vibegarb_db mysql -u vibegarb -p -e "SHOW TABLES;"

✅ Solução

1. Executar migrations:

# Executar todas as migrations
docker exec vibegarb_app php artisan migrate

# Se houver erro, forçar
docker exec vibegarb_app php artisan migrate --force

2. Verificar se tabelas foram criadas:

# Listar tabelas
docker exec vibegarb_db mysql -u vibegarb -p -e "USE vibegarb; SHOW TABLES;"

3. Executar seeders:

# Popular banco com dados
docker exec vibegarb_app php artisan db:seed
