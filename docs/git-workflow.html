Git Workflow

Fluxo de Branches e Controle de Versão

Atualizado: 07 de Julho de 2025

Visão Geral do Git Flow

O projeto Vibe Garb utiliza o Git Flow simplificado com duas branches principais para garantir estabilidade e facilitar o desenvolvimento colaborativo.

Branches Principais

🚀
main - Branch de produção (código estável e testado)
🔧
develop - Branch de desenvolvimento (funcionalidades em progresso)

Fluxo de Trabalho

1. Desenvolvimento Diário

1
Trabalhar na develop

Todo desenvolvimento acontece na branch develop

git checkout develop
2
Commits Frequentes

Fazer commits pequenos e descritivos regularmente

git add . && git commit -m "feat: nova funcionalidade"
3
Push para Origin

Enviar mudanças para o GitHub regularmente

git push origin develop

2. Release para Produção

1
Testes Completos

Validar todas as funcionalidades na develop

Verificar se tudo está funcionando
2
Merge para Main

Incorporar mudanças estáveis na main

git checkout main && git merge develop
3
Tag de Versão

Criar tag para marcar a versão

git tag -a v1.0.0 -m "MVP Completo"
4
Push Completo

Enviar main e tags para o GitHub

git push origin main --tags

Comandos Essenciais

Verificação de Status

# Ver branch atual e status
git status

# Ver todas as branches
git branch -a

# Ver histórico de commits
git log --oneline -10

Navegação entre Branches

# Ir para develop
git checkout develop

# Ir para main
git checkout main

# Criar nova branch (se necessário)
git checkout -b feature/nova-funcionalidade

Commits e Push

# Adicionar todas as mudanças
git add .

# Commit com mensagem descritiva
git commit -m "feat: implementar checkout completo"

# Push para branch atual
git push origin develop

# Push forçado (cuidado!)
git push origin develop --force

Merge e Release

# Merge develop para main
git checkout main
git merge develop

# Criar tag de versão
git tag -a v1.0.0 -m "Versão 1.0.0 - MVP Completo"

# Push da main com tags
git push origin main --tags

Convenções de Commit

Utilizamos o padrão Conventional Commits para manter o histórico organizado:

Tipos de Commit

✨
feat: Nova funcionalidade
🐛
fix: Correção de bug
📚
docs: Documentação
🎨
style: Formatação, CSS
♻️
refactor: Refatoração de código
🧪
test: Testes
🔧
chore: Configurações, build

Exemplos de Mensagens

# Boas práticas
feat: implementar sistema de checkout completo
fix: corrigir erro de validação no carrinho
docs: adicionar documentação da API Dimona
style: padronizar CSS da documentação
refactor: otimizar DimonaService
test: adicionar testes para OrderCreationService
chore: atualizar dependências do Docker

# Evitar
"ajustes"
"correções"
"update"
"changes"

Cenários Comuns

Cenário 1: Desenvolvimento Normal

# Trabalhar na develop
git checkout develop
git pull origin develop

# Fazer mudanças...
git add .
git commit -m "feat: nova funcionalidade"
git push origin develop

Cenário 2: Release para Produção

# Finalizar develop
git checkout develop
git add .
git commit -m "feat: MVP completo"
git push origin develop

# Merge para main
git checkout main
git pull origin main
git merge develop
git tag -a v1.0.0 -m "MVP Completo"
git push origin main --tags

# Voltar para develop
git checkout develop

Cenário 3: Hotfix em Produção

# Criar hotfix da main
git checkout main
git checkout -b hotfix/correcao-critica

# Fazer correção...
git add .
git commit -m "fix: correção crítica"

# Merge para main
git checkout main
git merge hotfix/correcao-critica
git tag -a v1.0.1 -m "Correção crítica"
git push origin main --tags

# Merge para develop também
git checkout develop
git merge hotfix/correcao-critica
git push origin develop

# Deletar branch hotfix
git branch -d hotfix/correcao-critica

Boas Práticas

✅ Fazer

  • Commits pequenos e frequentes
  • Mensagens descritivas usando convenções
  • Testar antes de fazer merge para main
  • Usar tags para versões importantes
  • Manter develop sempre atualizada
  • Fazer backup antes de operações complexas

❌ Evitar

  • Commits diretos na main
  • Mensagens genéricas ("update", "fix")
  • Commits gigantes com muitas mudanças
  • Push forçado na main
  • Trabalhar sem branch
  • Esquecer de fazer pull antes de push

Recuperação de Problemas

Problema: Commit Errado

# Desfazer último commit (mantém mudanças)
git reset --soft HEAD~1

# Desfazer último commit (remove mudanças)
git reset --hard HEAD~1

Problema: Merge Conflitos

# Ver arquivos em conflito
git status

# Editar arquivos manualmente
# Depois de resolver:
git add .
git commit -m "fix: resolver conflitos de merge"

Problema: Perder Mudanças

# Ver histórico de mudanças
git reflog

# Recuperar commit perdido
git checkout [hash-do-commit]
git checkout -b recuperacao
git checkout develop
git merge recuperacao

Resumo do Workflow

O Git Flow do Vibe Garb é simples e eficiente:

  • ✅ develop - Desenvolvimento contínuo
  • ✅ main - Código estável e testado
  • ✅ Tags - Versões importantes
  • ✅ Commits convencionais - Histórico organizado
  • ✅ Testes antes de merge - Qualidade garantida
